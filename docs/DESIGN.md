# wiki-eval — design rationale

This is the writeup for the wiki-eval prototype. It explains the choices behind the system, how the eval is structured, what the numbers say, and where the system breaks.

## What the system is

A Wikipedia-grounded Q&A agent built on the Anthropic API, with an eval harness that grades it on factual accuracy, faithfulness to Wikipedia, citation quality, refusal calibration, and false-premise handling.

- **Agent model:** `claude-sonnet-4-6` with two tools — `search_wikipedia` and `fetch_wikipedia_article` — backed directly by the live MediaWiki API. No hosted retrieval, no vector store, no scraping.
- **Judge model:** `claude-opus-4-7` with a structured 0–2 rubric returned as JSON.
- **Eval suite:** three dev datasets (82 cases) plus two heldout datasets (40 cases), all run with a closed-book baseline and an oracle reference for a 3-track comparison. 122 cases total.
- **Prompt iteration:** six versions (v1 → v6), four authored by hand and three (v3, v4, v5) generated by an auto-improver agent that has no eval access. **v3 wins** on average across dev + heldout (82.7% pass-rate) and is the CLI default.

The whole thing is ~1.5k lines of Python plus six Markdown prompt files.

## Why these models

- Sonnet 4.6 as the agent is the cost/quality sweet spot for tool-using agents. The tasks here are mostly retrieval + composition; Opus would be over-spec.
- Opus 4.7 as the judge because grading is a *harder* task than answering — the judge has to weigh agent answer vs. reference and distinguish "missing a key element" (acc=1) from "factually wrong on the main point" (acc=0). The cost is fine because we cache the judge's static system prompt with `cache_control: ephemeral` so repeated calls within an eval run are cheap.
- Opus 4.7 as the reference (Track B) for the same reason: it has to faithfully extract from gold article text without inventing.

## The agent loop

`src/wiki_eval/agent.py` runs the standard Anthropic tool-use loop:

1. Send the question with the `tools=` list.
2. While `stop_reason == "tool_use"`: dispatch each `tool_use` block locally, append the `tool_result` blocks to messages, repeat. Hard-capped at 15 turns (lifted from 10 after multi-constraint cases on bridges/movies repeatedly hit the cap).
3. When the model emits a `text` block as its final content, that's the answer.

The full trace (each tool call, args, result summary, latency, token counts) is preserved on `AgentRun` and serialized into the eval report. This lets the evaluation grade not just *what* the agent said but *how* it got there — `no_search` and `wrong_citation` are recoverable as failure modes only because we have the trace.

## Tools

Two tools, both thin wrappers over the MediaWiki API at `https://en.wikipedia.org/w/api.php`:

- `search_wikipedia(query, limit=5)` — `list=search` endpoint, returns titles + plaintext snippets.
- `fetch_wikipedia_article(title, max_chars=4000)` — `prop=extracts&explaintext=1`, with `redirects=1`. Caps the slice fed back to the model to protect the context window; the model can re-fetch with a larger `max_chars` (up to 12000) if it sees `[...article truncated]`.

The model gets clean prose, not wiki markup. Snippet HTML (`<span class="searchmatch">`) is stripped before the result is returned.

## Why three tracks

A single pass-rate for the agent is a useless number on its own. It conflates "the agent did good retrieval" with "the model already knew the answer from training data". The 3-track design separates these:

| Track | Setup | What it measures |
|---|---|---|
| **AGENT** | Sonnet 4.6 + Wikipedia tools | the system under test |
| **Track A** | Sonnet 4.6, *no tools, no internet, parametric only* | what the same model gets without retrieval |
| **Track B** | Opus 4.7 reading the gold articles (text pasted into the prompt, no tools either) | the oracle ceiling — what's achievable when the right articles are in hand |

The same judge rubric scores AGENT and Track A against Track B as the reference. The **AGENT − Track A** gap is the retrieval value. The **Track B − AGENT** gap is the headroom — what better retrieval or prompting could still recover.

This is why the cases are post-cutoff (see next section). If the Track A model already knows the answer, the gap collapses and you're just measuring the model.

### Closed-book really means closed-book

`closed_book_answer()` in `eval/judge.py` calls `client.messages.create(...)` *without* a `tools=` argument. The Anthropic API only enables tool use (including any hosted tools like web search) when `tools` is present, so the model is genuinely answering from parametric memory. The system prompt also forbids it from pretending to look anything up. The intent is documented inline so it can't drift:

```python
# Intentionally no `tools=` — closed-book means closed-book.
```

## The post-cutoff filter

The eval is intentionally biased toward content created after **2025-09-01** — Sonnet 4.6's knowledge cutoff is January 2026 in name but practical recall on events from late 2025 onward is patchy. By grounding cases in articles created (or in the case of bridges/movies, describing events occurring) after that date, Track A *should* fail. When it doesn't, the case is too easy and gets reworked.

Discovery is automated:

- `eval/find_post_cutoff_pages.py` — uses `srsort=create_timestamp_desc` on broad probes, then verifies each candidate's first-revision timestamp, then filters on incoming-link count and pageviews. Output is the candidate list that seeded the original 40 cases.
- `eval/discover_movies.py` + `eval/extract_movie_facts.py` — pulls recent film categories and parses release dates from the lead.
- `eval/discover_bridges.py` — walks `Category:Bridges completed in 2026` and its country sub-categories.
- `eval/heldout/discover_songs.py` — walks `Category:2025 songs`, `Category:2026 songs`, `Category:2025 singles`, `Category:2026 singles`; filters post-cutoff.
- `eval/heldout/discover_railway_lines.py` — walks `Category:Railway lines opened in 2026` (10 candidates: Line 5 Eglinton, Mumbai Red Line, Camp Hill line, Monon Corridor, Lausanne tramway, Line 17 São Paulo, Nanjing S2, Ningbo Line 6, GRU Airport People Mover, Jinhua–Jiande HSR).

Each script saves both a structured summary (`_summary.json`) and the full article text per candidate (`_*_corpus/*.txt`), so case authoring references real content instead of training-data recall. The heldout sets use the same methodology as the dev sets but were authored *after* prompt v6 was frozen and never inspected during prompt iteration.

## The judge rubric

`eval/judge.py` defines a 0–2 integer rubric over five dimensions:

| dimension | what 0 means | what 2 means |
|---|---|---|
| **accuracy** | factually wrong on the main point | fully correct |
| **faithfulness** | clearly hallucinates / invents details | every claim supported by the reference |
| **citations** | no inline `[Title]` and no Sources line | inline citations *and* Sources list naming actually-fetched articles |
| **refusal_calibration** | answered confidently when it should have refused (or refused when it shouldn't) | perfect calibration |
| **premise_handling** | accepted a false premise | corrected explicitly |

`overall_pass` is a hard conjunction: `accuracy ≥ 1 AND faithfulness == 2 AND citations ≥ 1 AND refusal_calibration ≥ 1 AND premise_handling ≥ 1`. Faithfulness gets the strict `== 2` because hallucination is the failure mode that destroys trust. The rest are gated at "passable" rather than "perfect" so high-quality answers with one missing citation still pass.

The judge also emits structured `failure_modes` tags (`hallucinated`, `missing_citation`, `wrong_fact`, `incomplete`, …). Aggregated counts in the report tell us *which lever to pull next* — see `.claude/skills/tune-prompt/SKILL.md` for the failure-mode → prompt-change mapping.

The judge's static system prompt (~2.5k tokens of rubric definition) is wrapped in `cache_control: {type: ephemeral}` so repeated calls within an eval run hit the cache. Track B's static system prompt is short enough not to need caching.

## The eval datasets we ship

Three dev datasets (82 cases) used to drive prompt iteration, plus two heldout datasets (40 cases) that were never inspected before prompts were frozen. 122 cases total.

| File | N | Role | Focus |
|---|---:|---|---|
| `test_cases.jsonl` | 40 | dev | broad post-cutoff facts: simple, multi-hop, temporal, false-premise, non-encyclopedic, disambiguation |
| `test_cases_movies.jsonl` | 22 | dev | 2025–26 films; multi-actor identity (find the film by overlapping cast) and semantic plot lookup, with grossing tie-breakers |
| `test_cases_bridges.jsonl` | 20 | dev | 2026 bridges; multi-constraint composition + longest/tallest/first/replacement tie-breaks + false premises |
| `eval/heldout/test_cases_songs.jsonl` | 20 | heldout | 2025–26 songs; multi-artist identity, semantic theme, chart tie-breaks, false premises |
| `eval/heldout/test_cases_railway_lines.jsonl` | 20 | heldout | 2026 railway lines; multi-constraint + comparative + tie-breaks + false premises |

Two further datasets (`test_cases_stubs.jsonl`, `test_cases_earthquake.jsonl`) were generated and run but are not part of the active suite — both saturated at ~100% AGENT pass on v1, so they don't discriminate between prompt versions. They remain as smoke tests.

Across all five active sets, **Track A pass% ≈ 0** (the post-cutoff filter is doing its job) and the AGENT − Track A gap is +60 to +85 points — a clean quantification of retrieval value.

## Prompt iteration: v1 → v6

We iterated six prompt versions, mixing hand-authored revisions with versions generated by an auto-improver agent. The single-change-per-version discipline made it possible to attribute regressions to specific rules.

| Version | Authored | Headline change |
|---|---|---|
| v1 | hand | Original — terse "ground claims in Wikipedia, cite what you fetched" |
| v2 | hand | Added: search-before-refuse, never-empty answer, tie-breaker enumeration, re-fetch on truncation, budget awareness |
| v3 | auto-improver | Added "Disambiguate before grounding" rule (covers ambiguous names like Mercury, Avatar) |
| v4 | auto-improver | Expanded false-premise into a two-step verification procedure |
| v5 | auto-improver | Added "Infobox facts" sub-rule for verbatim numeric/date values |
| v6 | hand | Added the verbatim-values rule + a new "Verify before answering" self-check section (5 sub-points: source-attribution, verbatim-value, sources/citations consistency, confidence calibration, refusal-mode self-check) |

### What v1 fails on (the failure-mode counts that drove iteration)

- `missing_citation` — most common across movies/bridges. The agent answers correctly but drops inline `[Title]` markers on multi-part answers.
- `incomplete` — multi-constraint questions: agent satisfies 2 of 3 constraints and stops.
- `no_search` — concentrated on the original 40 — 6 cases answered from parametric memory despite the explicit rule. Sonnet sometimes shortcuts on easy-looking questions.
- `wrong_fact` — small but persistent: agent fetches the right article, then misreads it. Often a default-truncation issue.
- `false_premise_accepted` — bridges has 2 cases that explicitly probe this. v1 took both at face value.

### Final results

Pass-rate (%), AGENT track, latest report per (dataset, prompt):

| Version | test_cases (dev) | movies (dev) | bridges (dev) | songs (HO) | rail (HO) |  avg |
|---|---:|---:|---:|---:|---:|---:|
| v1 | 87.5 | 63.6 | 65.0 | 80.0 | 75.0 | 74.2 |
| v2 | 92.5 | 72.7 | 70.0 | 85.0 | 55.0 | 75.0 |
| **v3** | **92.5** | **90.9** | **70.0** | **90.0** | **70.0** | **82.7** |
| v4 | 97.5 | 72.7 | 60.0 | – | – | 76.7 |
| v5 | 97.5 | 63.6 | 65.0 | – | – | 75.4 |
| v6 | 92.5 | 81.8 | 65.0 | 95.0 | 65.0 | 79.9 |

v4 and v5 weren't backfilled on heldout: both regressed on movies and bridges relative to v3, so the comparison was moot.

### What the numbers say

- **v3 is the clear winner.** It improves on v1 by **+8.5 pts on average** and wins on every dataset except a tie with v6 on `test_cases` and rail, where v6 is +5 and -5 respectively.
- **v3 → v4 → v5 illustrate a classic auto-improver failure mode.** v4 added a two-step false-premise procedure that paid off on `test_cases` (+5 → 97.5) but added enough verbosity / extra tool calls that movies dropped 18 pts and bridges 10 pts. v5 inherited v4's weight without the corresponding wins. The auto-improver is reading its own change-log; without eval access it can't see the regressions.
- **v6 helps direct-identification but hurts tie-breaks.** Songs jumps to 95% (best on any version) but railway lines drops to 65%. The tie-break-heavy questions need turn budget more than additional self-check overhead.
- **The heldout set behaves like the dev set.** v1 → v3 improvement on dev (+8.5) is matched by v1 → v3 on heldout (+5 on songs, -5 on rail; net wash, but no collapse). No evidence of dev-set overfitting — the rules are general.

## The auto-improver agent

`src/wiki_eval/improve_prompt.py` is a meta-agent (Opus 4.7) that iteratively revises the prompt **without seeing the eval data**. It takes the current prompt and a frozen failure-mode taxonomy (the same vocabulary the judge emits) and runs a critic → reviser cycle to produce the next version. v3, v4, and v5 came from this loop.

Two design decisions kept it honest:

1. **No eval access.** The improver cannot read `test_cases*.jsonl`, the gold articles, or any report. This was enforced by feedback memory after a near-miss where an early plan referenced specific case ids in the prompt — overfitting masquerading as iteration.
2. **One change per version.** The improver's revision instructions force a single named modification per output. This is what lets us read the v3-vs-v4 delta as "the two-step false-premise rule" rather than "many small edits".

The result: v3 is a real win, v4/v5 are real regressions, and we can attribute both to specific rules. An improver with eval access would have plateaued at v3 (rationally — its added rules were correct in isolation but interacted badly with the rest of the prompt).

## What the system can't do (and why that's fine)

- **No retrieval over content not in Wikipedia.** This was a constraint, not an oversight — Wikipedia is the source of truth by design. Questions that need primary sources (court filings, raw datasets, social media) fall back to refusal.
- **No real-time data.** "What's the score right now?" — Wikipedia has no current state. The system is told to refuse these.
- **No translation.** English Wikipedia only. Articles that exist on other Wikipedias and not en-wiki return `Article not found`.
- **No hierarchical retrieval.** A question requiring a chain of 4+ article hops will eventually exhaust the 10-turn budget. The current eval has 1- and 2-hop cases; 3-hop+ is the next class up.

## Performance lessons

A few infrastructure changes were forced by reality during the iteration:

- **HTTP timeouts on Anthropic clients.** Both `anthropic.Anthropic` instances (agent and judge) now set `timeout=120, max_retries=4`. Without an explicit timeout, a single stalled connection could hang the whole eval run for hours at 0% CPU — observed multiple times when running `workers ≥ 2` against songs/railway. With the timeout in place, stuck requests break loose and retry cleanly.
- **`closed_book_judge` caching.** The judge call that grades Track A's answer is deterministic per case (no agent dependency). It was being recomputed on every prompt version. Caching it cut ~30% of judge calls per backfill run.
- **Process-level parallelism beats thread-level on this workload.** Four parallel processes (one per `(prompt, file)` pair), each with `workers=4`, completes the 80-case heldout backfill in **3m34s** — a 16x speedup over `workers=1`. Per-process retry state is isolated, so a rate-limit hiccup in one file doesn't block the others.

## What I'd build next

Evaluation systems often fail because they test simple, single-hop questions. The first two items below are about closing that coverage gap; the rest are system-level improvements driven by what the existing evals already surface.

1. **Multi-hop questioning.** Add a dataset (or category in `test_cases.jsonl`) of cases that require linking information across two or more Wikipedia pages — e.g., *"What is the capital of the country where the inventor of the telephone was born?"* (Bell → Scotland → Edinburgh) or *"Which 2026 cable-stayed bridge was designed by the firm that also designed the longest 2025 cable-stayed bridge?"* The current suite has multi-constraint composition (one article, several attributes) but very little chained reasoning across articles. Multi-hop is where retrieval-vs-memory diverges most sharply, so it's also the regime where AGENT − Track A should look most favorable when the system works — and most embarrassing when it doesn't.
2. **Knowledge-conflict handling.** Wikipedia often has conflicting facts across sections, infoboxes, related articles, or older revisions (e.g., two snippets giving different dates for an event, or a country's GDP figure that disagrees between its main article and a "List of …" page). Add an evaluation loop that surfaces such cases and grades the system on detection + hedging, not just picking a side. Concretely: a `conflict_detected` failure mode in the judge rubric, a small dataset of curated conflict cases, and a prompt rule that says *"if two fetched articles disagree on a fact, name both values, attribute each, and say which (if either) the question's framing privileges."*
3. **Automated "golden dataset" generation.** The current suite tops out at 122 cases because each one is hand-authored. To scale evaluation into specific domains (medicine, law, geography, sports) without that bottleneck: use an LLM to scan a fetched Wikipedia article and emit **(question, answer, context)** triplets — the question grounded in the article, the answer extracted verbatim, the context the source span. Then run a second-pass LLM filter to drop ambiguous, leading, or trivially-paraphrased questions. Keep human review on a sampled slice (~5%) as a quality gate rather than the whole set. Treat synthetic cases as a separate dev fold and reserve a hand-authored heldout for final scoring — the pattern we already use, just with the dev fold scaled up 10–100x.
4. **A v7 that combines v3's strengths with v6's verbatim-value discipline,** without the verbosity that hurts tie-break-heavy datasets. The split between songs (+15 v6 vs v1) and rail (-10 v6 vs v1) on heldout is the lever. Likely shape: keep v3's body, add only the verbatim-values rule from v6, drop the 5-point self-check section.
5. **Tool affordance: a per-question turn budget hint.** Multi-hop / multi-constraint cases often plan well and then run out of turns. Letting the planner emit an "expected hop count" the model can reason about would help.
6. **An infobox-aware fetch tool.** A surprising fraction of the bridges/railways/earthquakes facts live in the infobox, which the plaintext extract preserves but doesn't structure. A second fetch mode that returns infobox key-values would shorten the path to a numerical answer.
7. **Evaluator-level cost tracking.** The report has token counts but no $ figure. A small post-processor would let us track "AGENT tokens per pass" as the optimization target alongside pass rate.
8. **A regression gate.** A test that runs the cached v1 numbers and fails CI if the headline pass rate drops. Right now nothing prevents a prompt change from silently breaking the original 40 cases.
9. **Auto-improver with read-only eval access on a held-back fold.** The current improver is *too* blind. A k-fold split where the improver sees k-1 folds' failure-mode counts (but not the cases or answers) might recover v3-class wins without v4/v5-class regressions.
10. **Passage-level judging (G-Eval / Prometheus style).** The current judge grades the *answer* on five dimensions; it doesn't grade the *retrieval* — i.e., whether each fetched article was actually relevant or actually supported the claim attributed to it. Add a second judge pass that, for each `[Article Title]` citation in the answer, scores the cited passage on **faithfulness** (does the article text support the claim?) and **relevance** (does the article actually address the question?). Use a G-Eval-style chain-of-thought rubric (the judge writes its reasoning before emitting the score) or a Prometheus-style direct-assessment rubric. This is what surfaces "looks-right but cites the wrong article" failures that the current end-to-end judge sometimes misses, and it gives the prompt-tuning loop a finer-grained signal than a single `wrong_citation` tag.
